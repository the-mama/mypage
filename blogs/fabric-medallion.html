Medallion Architecture in Microsoft Fabric

Implementing the three-layer architecture for enterprise analytics using Fabric's OneLake and Delta Lake

May 2024 • Microsoft Fabric, OneLake, Architecture

Why Medallion in Fabric?

Microsoft Fabric is built on Delta Lake. The medallion architecture maps perfectly to Fabric's lakehouse model. Unlike traditional data warehouses, Fabric abstracts away infrastructure—you focus on data logic.

The Three Layers

Bronze Layer: Raw Data Landing

Store raw data exactly as received from source systems. No transformations, no schema enforcement. This is your immutable source of truth for auditing and compliance.

  • Data ingested via Data Factory pipelines or direct uploads
  • Stored in OneLake (Fabric's unified storage)
  • Partitioned by ingestion date for lifecycle management
  • Append-only: new data added, old data never deleted
  • Example: Raw JSON from APIs, CSV uploads, database exports

Silver Layer: Cleaned & Validated

Apply business rules, deduplication, and schema validation. This is where data governance begins. Schema is enforced. Quality checks are applied.

  • Remove duplicates using natural key + timestamp
  • Type casting: dates to DATE, amounts to DECIMAL
  • Standardization: trim whitespace, uppercase/lowercase
  • Null handling: replace invalid NULLs with defaults
  • Add audit columns: load_timestamp, source_system

Gold Layer: Business-Ready Analytics

Pre-aggregated tables optimized for BI and analytics. Power BI and other tools query this layer directly.

  • Domain-specific marts: Sales, Finance, Marketing, Operations
  • Denormalized star schema for speed
  • Pre-calculated metrics and KPIs
  • Materialized as tables (not views) for performance

Raw customer transactions stored as Parquet files, partitioned by date:

Silver Layer: Transformation in Notebook

from pyspark.sql import SparkSession
from pyspark.sql.functions import col, current_timestamp
from delta.tables import DeltaTable

spark = SparkSession.builder.appName("medallion").getOrCreate()

# Read Bronze
bronze_df = spark.read.format("delta")\
  .load("/mnt/bronze/transactions")

# Silver transformations
silver_df = bronze_df \
  .filter(col("transaction_id").isNotNull()) \
  .filter((col("amount") > 0) & (col("amount") < 1000000)) \
  .dropDuplicates(["transaction_id"]) \
  .withColumn("transaction_date", col("transaction_date").cast("date")) \
  .withColumn("amount", col("amount").cast("decimal(10,2)")) \
  .withColumn("loaded_at", current_timestamp())

# Upsert into Silver
silver_table = DeltaTable.forPath(spark, "/mnt/silver/transactions")
silver_table.alias("t").merge(
  silver_df.alias("s"),
  "t.transaction_id = s.transaction_id"
).whenMatchedUpdateAll() \
 .whenNotMatchedInsertAll() \
 .execute()

print("Silver layer updated successfully")

Gold Layer: SQL Aggregations

-- Create daily transaction summary
CREATE TABLE gold.daily_transaction_summary AS
SELECT
  DATE(transaction_date) as transaction_date,
  customer_id,
  COUNT(*) as transaction_count,
  SUM(amount) as total_amount,
  AVG(amount) as avg_amount,
  MIN(amount) as min_amount,
  MAX(amount) as max_amount
FROM silver.transactions
GROUP BY DATE(transaction_date), customer_id;

Fabric-Specific Features

Use Fabric Pipelines for Orchestration

Schedule notebooks with built-in scheduler. Monitor execution with visual DAG.

Enable Row-Level Security (Gold Layer)

Control which users see which data. Apply at the table level, then connect to Power BI.

Use Shortcuts Instead of Copying

Create shortcuts to Bronze tables from other workspaces. Data lives once, referenced many times.

Monitor with Real-Time Analytics

Use Fabric's KQL database to track pipeline execution, data quality metrics, and data freshness.

Cost Optimization

  • Compress Bronze: Parquet with snappy compression saves 70% storage
  • Partition by date: Query only relevant dates. Scan cost drops 90%
  • Use Delta cache: Fabric auto-caches hot tables. Subsequent queries are instant
  • Archive old data: Move Bronze data older than 1 year to archive storage
  • Minimize shuffles: SQL wins over Spark for aggregations (better query optimization)

Real-World Scale

A retail company implemented medallion for 50M+ daily transactions:

  • Bronze: 500GB raw daily data from POS, online, inventory systems
  • Silver: Deduplicated, cleansed (150GB after compression)
  • Gold: 6 analytics marts (store-level, product, customer, supply chain, financial, KPI)
  • Pipeline runtime: 30 minutes end-to-end (Bronze → Silver → Gold)
  • Cost: $50 per day on F64 Fabric capacity (covers 500+ daily workloads)
  • Availability: 99.9% uptime (managed by Microsoft)

Key Takeaways

  • Medallion is not a suggestion—it's the standard for modern data platforms
  • Bronze is immutable. Silver is cleansed. Gold is optimized for BI
  • Use notebooks for Bronze → Silver. Use SQL for Silver → Gold (better performance)
  • Partition everything by date. Enables retention policies and cost control
  • Monitor data freshness and quality. Automate alerts for stale or bad data

Fabric's managed infrastructure removes the pain of cluster management. Medallion architecture removes the pain of data organization. Together, they enable teams to focus on analytics, not operations.
