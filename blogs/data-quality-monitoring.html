Data Quality Monitoring at Scale

Building automated, cost-effective DQ systems that catch issues before they reach your users

May 2024 • Data Quality, Great Expectations, Monitoring

The Problem: Why DQ Matters

Bad data costs money. A simple example: a data warehouse team didn't catch that a customer ID field was being NULL'd out upstream. The BI team built dashboards on it. Sales reported inflated numbers. Investors questioned the growth narrative. Cost: 3 months to rebuild, revalidate, and apologize.

Manual testing doesn't scale. You can't inspect 100M rows by hand. You need automated, intelligent quality gates.

Data Quality Dimensions

Before building a DQ system, understand what you're monitoring:

  • Completeness: Are all required fields populated? NULL or empty rate < threshold?
  • Accuracy: Do values match the source of truth? (e.g., revenue from accounting system)
  • Consistency: Do related fields agree? (customer name in orders table vs. customers table)
  • Timeliness: Is data fresh? Last update timestamp within SLA?
  • Uniqueness: Are there unexpected duplicates? PK violations?

My Implementation: 3-Tier Approach

Tier 1: Automated Schema Validation

Run on every pipeline execution. Catches obvious breaks immediately.

SELECT
  column_name,
  data_type,
  is_nullable,
  COUNT(*) as NULL_count
FROM my_table
WHERE 1=1
GROUP BY column_name, data_type, is_nullable
HAVING is_nullable = 0 AND NULL_count > 0

This query alerts you if a NOT NULL column has NULLs (schema contract violation).

Tier 2: Statistical Profiling (Weekly)

Compare current metrics against baseline. Detect drift early.

  • Min/max/median for numeric columns
  • Distribution histograms (detect new value categories)
  • Cardinality trends (is the distinct count exploding?)
  • Percent complete by column

Tier 3: Business Logic Tests (On-Demand)

Domain-specific assertions. These need human input to define.

  • Revenue always > 0 (no negative sales allowed)
  • Order quantity matches invoice quantity
  • Customer age 18–120 (catch data entry mistakes)
  • No future dates unless explicitly allowed

Cost Optimization Tricks

Sample, Don't Scan

For 1B-row tables, a 1% random sample is statistically significant for most tests. This cuts scanning costs by 99% while maintaining quality. 

SELECT *
FROM my_huge_table
WHERE RAND() < 0.01
LIMIT 10000000

Incremental Quality Checks

Don't recheck all historical data. Only validate new/changed rows. 

-- Only check rows loaded today
SELECT *
FROM silver.transactions
WHERE dbt_loaded_at::DATE = CURRENT_DATE

Aggregate First, Check After

Test aggregated metrics (daily revenue sum) not raw rows. Much cheaper.

Real Incident: What DQ Prevented

A payment processor's transaction feed started sending duplicate records. Without quality checks, analysts would have reported double the transaction volume.

Our uniqueness check caught it within 2 minutes: duplicate transaction IDs exceeded threshold. Pipeline auto-paused. Engineering team investigated and found a race condition in the source system's write logic. Fixed in 30 minutes. Impact: zero.

Recommended Tools

  • Great Expectations: Python-native DQ framework. Integrates with dbt, Spark, Databricks.
  • dbt Tests: Version-controlled, simple, integrated with your data model.
  • Custom SQL + Airflow: Full control. Overkill unless you have complex custom logic.
  • Databricks Quality Monitoring: Built-in, easy for Delta Lake tables.

Key Takeaways

  • Quality checks are not overhead—they're insurance against costly mistakes
  • Start simple (NOT NULL, uniqueness). Add complexity as needed
  • Automate tier 1 & 2. Keep tier 3 small and business-driven
  • Sample aggressively. Most tests don't need 100% of data
  • Alert on change, not on absolute value. "Revenue is odd" matters more than "Revenue is $1M"

Bad data is preventable. Invest in quality systems early. Your future self will thank you.
