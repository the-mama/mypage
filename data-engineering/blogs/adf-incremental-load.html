Azure Data Factory: Incremental Load Pattern

Production-grade incremental loading with watermark strategy — scaling from millions to billions of rows

June 2024 • Azure Data Factory, ETL, Production

The Problem

In real-world data engineering systems, loading full datasets every time is not scalable. Imagine a source system with 500M rows. A full reload takes 3 hours, costs $200, and crushes the network. You need to ingest only new or modified data—incrementally.

The Solution: Watermark-Based Incremental Load

A watermark is a marker that tracks the last successful load. It answers the question: "What was the last timestamp/ID I loaded?"

Architecture Design

  • Control Table: Stores the watermark (last_load_date, last_load_id)
  • Source System: SQL database or API with a ModifiedDate column
  • ADF Lookup Activity: Retrieves the watermark from the control table
  • Copy Activity: Runs a parameterized query using the watermark
  • Sink: ADLS Gen2 (Bronze Layer) or SQL DW
  • Update Control Table: After successful load, update the watermark

Implementation: Step by Step

Step 1: Create the Control Table

CREATE TABLE dbo.WatermarkTable (
    WatermarkID INT PRIMARY KEY,
    TableName NVARCHAR(255),
    WatermarkColumn NVARCHAR(255),
    WatermarkValue DATETIME,
    LastUpdateTime DATETIME
);

INSERT INTO dbo.WatermarkTable VALUES
(1, 'SourceTable', 'ModifiedDate', '2024-01-01', GETDATE());

Step 2: ADF Pipeline Configuration

  • Lookup Activity 1: Read current watermark from control table
  • Lookup Activity 2: Get MAX(ModifiedDate) from source
  • Copy Activity: Extract only rows WHERE ModifiedDate > @WatermarkValue
  • Stored Procedure Activity: Update control table with new watermark

Step 3: Parameterized Query

SELECT *
FROM SourceTable
WHERE ModifiedDate > @LastWatermark
  AND ModifiedDate <= @NewWatermark
ORDER BY ModifiedDate;

Key Considerations

Late-Arriving Data

Records sometimes arrive late due to system delays. Add a buffer window to avoid missing data. 

-- Load data from watermark to (now - 5 minutes)
WHERE ModifiedDate > @LastWatermark
  AND ModifiedDate <= DATEADD(MINUTE, -5, GETDATE())

Handling Updates vs. Inserts

If your source tracks both inserts and updates with ModifiedDate, this pattern handles both automatically. If you only have inserts, use an CreatedDate column instead.

Idempotency

What if the pipeline fails after copying but before updating the watermark? The next run will re-load the same data. Make your sink idempotent:

  • Use MERGE in SQL to upsert data
  • Or use Parquet/Delta files with OVERWRITE mode for specific date partitions

Real-World Example

At a fintech company, we loaded daily transaction records (100M+) into ADLS.

  • Without incremental: 2.5 hours per day, $150 compute cost
  • With watermark: 8 minutes per day, $5 compute cost
  • Savings: 96% time reduction, 97% cost reduction

Benefits

  • Reduced data movement by 95%+ for daily loads
  • Faster pipeline execution (minutes instead of hours)
  • Lower infrastructure costs
  • Supports near real-time ingestion (can run hourly or more frequently)
  • Scales to billions of rows without overhead

Lessons Learned

  • Always add a buffer window (5–10 minutes) for late data
  • Monitor for watermark drift—if data arrives out of order, you'll miss it
  • Test failure scenarios: What happens if the pipeline crashes mid-run?
  • Log every watermark update—it saves hours during debugging
  • Use source freshness checks: Alert if no new data arrives for N hours (data quality gate)

Watermark-based incremental loading is the industry standard for a reason. Once you implement it once, you'll use it forever.
