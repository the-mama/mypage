Data Engineering

Slowly Changing Dimensions Type 2 (SCD Type 2)

Track complete historical changes in dimension tables — the Kimball standard for data warehouses

April 2024 • SQL, Data Warehouse, Kimball

The Business Problem

Your customer's address changes. Did they move, or was it a data entry error? You need to answer: "What was this customer's address on January 15, 2024?" Without SCD Type 2, you can't. You overwrote the old address.

SCD Type 2 solves this by keeping all versions of the record and marking which one is current.

The Three SCD Types (Quick Reference)

  • Type 1: Overwrite old value (no history)
  • Type 2: Keep all versions with effective dates (full history)
  • Type 3: Keep current + previous value only

Type 2 is most common in enterprise data warehouses.

Table Design

CREATE TABLE dim_customer (
    customer_key INT PRIMARY KEY,        -- Surrogate key
    customer_id INT,                     -- Natural key (from source)
    name NVARCHAR(255),
    email NVARCHAR(255),
    address NVARCHAR(500),
    city NVARCHAR(100),
    effective_start_date DATE,           -- When this version became active
    effective_end_date DATE,             -- When this version expired (NULL = current)
    is_current BIT,                      -- 1 = current, 0 = historical
    dbt_loaded_at TIMESTAMP              -- Audit timestamp
);

CREATE INDEX idx_customer_id_current
ON dim_customer(customer_id, is_current);

Load Logic: Step by Step

Step 1: Find Changed Records

Compare incoming data against the current version in the dimension.

SELECT
    src.customer_id,
    src.name,
    src.email,
    src.address,
    src.city
FROM staging_customer src
LEFT JOIN dim_customer dim
    ON src.customer_id = dim.customer_id
    AND dim.is_current = 1
WHERE src.name != dim.name
   OR src.email != dim.email
   OR src.address != dim.address
   OR src.city != dim.city
   OR dim.customer_key IS NULL  -- New customer

Step 2: Expire Old Records

Mark existing records as historical.

UPDATE dim_customer
SET effective_end_date = CAST(GETDATE() AS DATE),
    is_current = 0
WHERE customer_id IN (
    SELECT customer_id FROM changed_records
)
AND is_current = 1;

Step 3: Insert New Records

Add the new version with effective dates.

INSERT INTO dim_customer (
    customer_id, name, email, address, city,
    effective_start_date, effective_end_date, is_current,
    dbt_loaded_at
)
SELECT
    customer_id, name, email, address, city,
    CAST(GETDATE() AS DATE) as effective_start_date,
    NULL as effective_end_date,
    1 as is_current,
    GETDATE() as dbt_loaded_at
FROM staging_customer;

Querying Historical Data

Get the Current Address

SELECT name, address
FROM dim_customer
WHERE customer_id = 12345 AND is_current = 1;

Get Historical Timeline

SELECT
    effective_start_date,
    effective_end_date,
    address,
    ROW_NUMBER() OVER (ORDER BY effective_start_date) as version
FROM dim_customer
WHERE customer_id = 12345
ORDER BY effective_start_date;

Point-in-Time Query ("What was the address on a specific date?")

SELECT name, address
FROM dim_customer
WHERE customer_id = 12345
  AND effective_start_date <= '2024-01-15'
  AND (effective_end_date IS NULL OR effective_end_date > '2024-01-15');

Common Mistakes (and How to Avoid Them)

Mistake 1: Not Handling NULLs

src.address != dim.address returns NULL when either is NULL. Use COALESCE: 

WHERE COALESCE(src.address, '') != COALESCE(dim.address, '')

Mistake 2: Missing the Natural Key in the WHERE Clause

Always filter by natural key (customer_id) when joining. Otherwise you'll match the wrong customer.

Mistake 3: Not Indexing Correctly

Index on (natural_key, is_current) for fast lookups. Also index on effective_start_date for range queries.

Mistake 4: Forgetting to Handle Non-Changing Attributes

If customer name changes but email doesn't, you still need to create a new row. Every change triggers a new record.

Real-World Example

A fintech company tracked customer KYC (Know Your Customer) status updates:

  • Jan 1: Customer verified (status = 'VERIFIED')
  • Mar 15: Address updated during fraud check (status = 'VERIFIED', address changed)
  • Jun 20: Status revoked (status = 'SUSPENDED')

With SCD Type 2, they could answer: "Was this customer verified on April 1?" (Yes). "How long were they verified?" (Jan 1 to Jun 20). "What was their address during the fraud investigation?" (The mid-update address).

Performance Tips

  • Batch loads: Load hourly or daily, not row-by-row
  • Archive old records: Move records older than 3 years to a history table
  • Use staging tables: Load into staging first, then merge. Atomic updates.
  • Cluster by natural key: Speeds up lookups and updates

Key Takeaways

  • SCD Type 2 keeps full history. Use it for compliance, analytics, and auditability
  • Always use effective dates (start and end). They enable point-in-time queries
  • Use a surrogate key as primary key, not the natural key
  • Index on natural key + is_current for fast current lookups
  • This pattern scales to billions of rows if designed correctly

Every data warehouse needs SCD Type 2 for dimension tables. Learn it once, use it everywhere.
